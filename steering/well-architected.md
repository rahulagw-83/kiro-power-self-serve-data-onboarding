# AWS Well-Architected Framework — Enforcement Rules

> **Always-on standard.** Every pipeline generated by this power MUST satisfy these rules.
> Engineers don't memorize 44 rules — Kiro reads this file and enforces them automatically.

## Pillar 1: Security

| # | Rule | Enforcement |
|---|------|-------------|
| SEC-01 | Credentials MUST be stored in AWS Secrets Manager | Pipeline rejected if hardcoded |
| SEC-02 | Generated code MUST NOT contain any credential, token, or key | Pipeline rejected |
| SEC-03 | IAM roles MUST follow least-privilege (scoped per pipeline, not admin) | CDK generates scoped roles |
| SEC-04 | All JDBC/API connections MUST use SSL/TLS | Pipeline rejected |
| SEC-05 | S3 buckets MUST have SSE-S3 or SSE-KMS encryption | CDK enforces |
| SEC-06 | S3 buckets MUST block public access | CDK enforces |
| SEC-07 | PII/PHI columns MUST be tagged in Lake Formation | Pipeline rejected if PII without tag |
| SEC-08 | De-identification (hash/mask/redact) MUST be applied to PII before writing | Pipeline rejected if PII without deid_strategy |
| SEC-09 | Secrets MUST have rotation enabled (90 days max) | Warning + recommendation |
| SEC-10 | CloudTrail MUST be enabled for API audit | Deployment gate |
| SEC-11 | VPC Security Groups MUST allow only required ports (e.g. 5432 for PG) | Connection setup validates |
| SEC-12 | Glue job logs MUST NOT contain credential values | Structured logging masks secrets |

### Which steering files enforce this:
- `security-and-access.md` — IAM patterns, approval matrix, Lake Formation
- `connection-setup.md` — Secrets Manager, VPC security groups
- `ingestion-standards.md` — No-credential-in-code rule

## Pillar 2: Reliability

| # | Rule | Enforcement |
|---|------|-------------|
| REL-01 | Every pipeline MUST have retry logic (3 retries, exponential backoff) | Step Functions ASL generated with retries |
| REL-02 | Failed rows MUST go to quarantine (never dropped silently) | Glue job routes to _quarantine table |
| REL-03 | Pipelines MUST be idempotent (re-run produces same result) | SHA-256 dedup + MERGE INTO |
| REL-04 | Pre-ingestion MUST validate: connectivity, credentials, freshness, target, contract | 5-point pre-check in Step Functions |
| REL-05 | Post-ingestion MUST validate: row count, quarantine %, SLA, metrics | Post-check Lambda/step |
| REL-06 | Critical failure (>5% quarantine) MUST halt pipeline + alert | SNS + Step Functions fail state |
| REL-07 | Cross-batch dedup MUST use Iceberg MERGE INTO | Generated in Glue job |
| REL-08 | Schema drift MUST be detected and classified (low/medium/high) | Schema inference module |
| REL-09 | Breaking schema changes MUST pause pipeline + escalate | Alert + manual review gate |
| REL-10 | Iceberg snapshots MUST be enabled for time-travel recovery | Table properties in CDK |

### Which steering files enforce this:
- `ingestion-standards.md` — dedup, quarantine, idempotency
- `onboarding-workflow.md` — pre/post checks in workflow
- `schema-inference.md` — drift detection and handling
- `troubleshooting.md` — recovery procedures

## How Enforcement Works

```
Engineer writes YAML config (onboarding-request + pipeline-config)
    │
    ▼
Kiro loads POWER.md + all 'inclusion: always' steering files
    │
    ▼
For EACH generated artifact, Kiro validates against:
    ├── SEC-01 to SEC-12   (Security)
    ├── REL-01 to REL-10   (Reliability)
    ├── OPS-01 to OPS-10   (Operational Excellence)
    ├── PERF-01 to PERF-08 (Performance)
    ├── COST-01 to COST-08 (Cost)
    └── SUS-01 to SUS-04   (Sustainability)
    │
    ▼
ALL pass → generate compliant code
ANY fail → flag violation, request fix or justification
```
